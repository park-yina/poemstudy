# AdminLifecyclePolicy.md

## 📌 문제

매장 폐점 기능 추가 과정에서,
해당 매장의 관리자 계정을 어떻게 처리할 것인지 고민이 발생하였다.

초기에는:

- 관리자 계정 삭제
- temp 계정 상태 전환

방식을 고려하였다.

---

## ⚠ 문제점

### 계정 삭제

- 관리자 이력 및 로그 데이터 손실
- 재오픈 시 계정 복구 불가능
- 운영 데이터 단절

---

### temp 상태 재활용

기존 temp 상태는:

초기 비밀번호 변경 유도

를 의미하고 있었다.

하지만 폐점 상태까지 함께 표현하기 시작하면:

TEMP ≠ INACTIVE

상태 의미가 혼재되기 시작하였다.

---

# 🎯 결정

관리자 계정을 삭제하지 않고:

ACTIVE / INACTIVE

상태를 분리하여 운영하기로 결정하였다.

폐점은 “데이터 삭제”가 아니라:

운영 중단 상태

로 정의하였다.

---

# 🛠 적용

- `AdminStatus` enum 추가
- 폐점 시 관리자 자동 비활성화
- 인증 단계에서 상태 검증 추가
- 재오픈 시 관리자 상태 복구 가능하도록 설계

---

# 📦 결과

- 관리자 이력 보존 가능
- 운영 정책 의미 분리
- 상태 기반 인증 구조 완성
- 재오픈 대응 가능
- 운영 Lifecycle 확장 가능

---

# 💡 회고

이번 작업은 단순 기능 구현보다:

운영 상태를 데이터 모델로 어떻게 표현할 것인가

에 가까운 문제였다.

특히:

- 상태 의미 분리
- 데이터 보존
- 인증 정책 연동
- 운영 Lifecycle

을 함께 고려하면서,
운영 중심 관리자 구조를 설계할 수 있었다.
